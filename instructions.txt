PROJECT OVERVIEW
═══════════════════════════════════════════════════════

Project Name: AEGIS (Adaptive FPGA-Based Side-Channel Defense 
with Neural Attack Resilience)

Hardware: Digilent Arty S7 (Xilinx Spartan-7 XC7S50)
HDL: Verilog (not SystemVerilog unless I explicitly ask)
FPGA Toolchain: Vivado 2025.2 primary, F4PGA/Yosys as secondary 
               (for open-source EDA bonus point)
Host OS: Windows 11
Python Environment: Python 3.11.0 inside a virtual environment (venv)
                    Libraries: numpy, torch, torchvision, matplotlib,
                               pyserial, scipy, scikit-learn, tqdm
Clock: 100MHz system clock (Arty S7 default)
UART: 9600 baud, 8N1

The project has three acts:

ACT 1 — Build and Break:
  - AES-128 core in Verilog (iterative, not fully unrolled)
  - Hamming Weight power trace simulation
  - UART interface for PC communication
  - Classical DPA attack in Python

ACT 2 — The Clever Attack:
  - Boolean masking countermeasure on AES core
  - Neural network (MLP in PyTorch) breaks the masking
  - Key Rank analysis comparing DPA vs neural network

ACT 3 — The Real Defense:
  - Ring oscillator TRNG from Spartan-7 primitives
  - Randomized inter-round timing jitter
  - Continuous mask refreshing from TRNG
  - Both attacks demonstrated failing against hardened design

HARDWARE RULES — ALWAYS FOLLOW THESE
═══════════════════════════════════════════════════════

1. TARGET DEVICE: Always write for Xilinx Spartan-7 XC7S50. 
   Never give generic Verilog without considering synthesis 
   implications on this specific device.

2. RING OSCILLATOR PROTECTION: Any ring oscillator code must 
   always include both:
   (* KEEP = "TRUE" *) on all oscillator nets
   (* DONT_TOUCH = "TRUE" *) on all oscillator instances
   Never omit these. Vivado will silently optimize them away 
   without these attributes.

3. PBLOCK CONSTRAINTS: When writing ring oscillator code, 
   always include the XDC Pblock constraint that places 
   oscillators in SLICE_X0Y0:SLICE_X3Y3 to maximize 
   thermal entropy coupling.

4. LUT BUDGET AWARENESS: The XC7S50 has 32,600 LUTs. 
   My project uses roughly:
   - AES core: ~4,000 LUTs
   - Masking logic: ~2,000 LUTs
   - TRNG: ~500 LUTs
   - UART + control: ~500 LUTs
   Total: ~7,000 LUTs (well within budget)
   Do not suggest fully unrolled AES pipelines — they waste 
   LUTs with no benefit for this project.

5. ITERATIVE AES ARCHITECTURE: The AES core must use an 
   iterative round architecture with an FSM, not a fully 
   unrolled pipeline. States: IDLE → INIT → ROUND → FINAL → DONE

6. COLUMN-MAJOR STATE ORDERING: The 128-bit AES state is 
   stored column-major:
   bits [127:120] = row0, col0
   bits [119:112] = row1, col0
   bits [111:104] = row2, col0
   bits [103:96]  = row3, col0
   bits [95:88]   = row0, col1
   ... and so on
   Always use this ordering consistently across all modules.

7. SYNCHRONOUS RESET: All sequential modules must use 
   synchronous reset, active high, on the 100MHz clock.

8. NO ASYNCHRONOUS LOGIC: Never use initial blocks for 
   synthesis — only for testbenches. Never use #delays 
   in synthesizable code.

SECURITY / CORRECTNESS RULES
═══════════════════════════════════════════════════════

1. MASKING DISCIPLINE: In any masked AES module, the 
   unmasked intermediate value must never appear as a named 
   signal. If you write code where D_unmasked is a wire, 
   flag it immediately — that defeats the purpose of masking.

2. TRNG VALIDATION: Any TRNG code must include the entropy 
   test that verifies the ones ratio falls between 45%-55% 
   over 10,000 bits before the output is used by the 
   masking logic.

3. SEPARATION OF CONCERNS: Keep three separate top-level 
   designs:
   - aes_vulnerable.v    (no countermeasures, for Act 1)
   - aes_masked.v        (masking only, for Act 2)
   - aes_hardened.v      (masking + TRNG timing, for Act 3)
   Never mix countermeasure logic into the vulnerable version.

4. ATTACK INTEGRITY: DPA and neural network attacks must 
   target only byte 0 of the key for demonstration purposes. 
   Full 16-byte recovery is out of scope.
